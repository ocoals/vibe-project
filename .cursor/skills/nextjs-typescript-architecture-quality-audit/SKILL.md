# Next.js + TypeScript 아키텍처·코드 품질 감사

## 역할

시니어 소프트웨어 아키텍트 관점에서 **요청된 범위의 파일·경로**만 검토한다. **코드 수정은 하지 않고** 표 형식 리포트만 출력한다.

## 감사 항목 (필수)

1. **코드 중복**: 동일·유사 로직이 여러 파일에 산재하는지(유효성 검사, fetch·에러 파싱, 환경 변수 읽기, Supabase 호출 패턴 등).
2. **함수·컴포넌트 크기**: 단일 파일·함수·컴포넌트가 **대략 50줄을 넘어** 여러 책임(폼·목록·모달·API 호출 등)을 한데 묶는지. **단일 책임 원칙(SRP)** 위반 여부를 서술적으로 판단한다(줄 수는 경고 기준이며 절대 기준은 아님).
3. **타입 정의**: `any` 사용, `unknown` 이후 무분별한 단언, DB/API 행 타입의 `unknown`·과도한 유니온, 공개 API에 대한 인터페이스 누락 등 **느슨한 타입**과 개선점.
4. **네이밍 일관성**: 함수·변수·React 컴포넌트·파일명이 프로젝트 관례(예: 컴포넌트 PascalCase, 유틸 camelCase, 라우트·파일 kebab-case)와 어긋나는지.
5. **불필요한 의존성**: 사용하지 않는 import, 죽은 코드로 이어지는 props, **과도한 props drilling**(중간 컴포넌트가 전달만 하며 실제로는 하위에서만 필요한 데이터) 여부.

## 우선순위

- **High**: 유지보수·버그 위험이 크거나, 타입 안전성이 실질적으로 깨지는 패턴, 중복으로 인한 수정 누락 위험이 큰 경우.
- **Medium**: 개선 시 이득이 분명하나 긴급도는 낮은 경우, 일정 수준의 중복·느슨한 타입.
- **Low**: 스타일·미세한 일관성, 규모가 작아 영향이 제한적인 경우, “나중에 정리” 수준의 제안.

## 출력 형식 (필수)

아래 **한 개의 마크다운 표**로만 정리한다. 해당 사항이 없으면 한 행으로 「검토 범위 내 해당 없음」 또는 항목별 N/A를 적는다.

```markdown
| 일련번호 | 우선순위 | 파일 | 위치 | 문제 | 권고사항 |
|---|---|---|---|---|---|
```

- **위치**: 대략적인 줄 번호, 또는 섹션(예: `handleSubmit`, `POST 핸들러`, `자식 컴포넌트 props`).
- **문제**: 관찰된 사실을 짧게.
- **권고사항**: 구체적이고 실행 가능한 정리 방향(공통 훅·유틸 분리, 컴포넌트 분할, 타입 좁히기 등). **코드 블록으로 수정안을 작성하지 않는다**(설명만).

## 절차

1. 사용자가 지정한 파일·디렉터리만 연다. `.env` 등 비밀 값이 들어갈 수 있는 파일의 **실제 값은 읽지 않는다**.
2. 위 다섯 항목을 순서대로 체크한다. App Router·서버/클라이언트 경계(`"use client"`)가 있으면 역할 혼선이 없는지 참고한다.
3. **성능·보안 전용 감사**가 목적이면 각각 전용 스킬을 쓰도록 안내할 수 있다(이 스킬은 구조·중복·타입·네이밍 중심).
4. 발견이 없어도 항목별로 **N/A** 행을 남길 수 있다.

## 금지

- 사용자가 요청하지 않은 대규모 리팩터·자동 수정을 기본으로 하지 않는다.
- 실제 비밀 값을 리포트에 적지 않는다.
